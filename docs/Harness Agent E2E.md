## 端到端自主：跨越自制阈值

当智能体的自主能力不断飙升，到底它生成的范围可以扩展到哪一步？一旦跨越了某个工程临界点，智能体将完全接管，从代码到运维的每一环，是高度自治、端到端的全流程驱动。

本期我们继续精读OpenAI工程博客《Harness Engineering》，看AI开发的自主性究竟可以到达何种程度。

### 重新界定智能体生成的范围

首先要重新界定智能体生成的范围。在OpenAI的这个实验中，当他们说这个代码库是由Codex生成时，指的不只是：
- 产品代码和测试代码
- 它还包含CI配置与发布工具链
- 内部的开发管理工具
- 架构设计的历史文档
- 评估系统
- 测试的框架
- PR下面的审查评论与回复
- 管理仓库本身的脚本
- 生产环境的监控面板定义文件

并且智能体正在像人类一样直接使用标准的开发工具——他们会拉取审查意见，进行内联回复、推送更新，甚至经常自己压缩并合并PR。也就是说，他正在深度参与并运转着整个工程系统。

### 人类角色的转变

当繁琐的执行环节被全面接管，人类工程师在干什么？

人类依然在回路中，但工作重心已经转移到了一个完全不同的抽象层。人类不再是写具体代码的工人，而是成为了**定义规范的甲方工程师**。

现在的核心工作是三件事儿：
1. 排定优先级
2. 将用户反馈翻译成明确的验收标准
3. 最终验证交付成果

如果在中间过程中智能体卡壳了，人类也绝对不会下场去帮他逐行修代码。他们会把智能体的挣扎视为一种报错信号，去排查是不是系统里少了某个检测工具、缺了某种护栏，还是文档没写清楚。找到缺口并补充到系统后，依旧让智能体自己去完成代码修复。

**人类不再是执行者，而是环境与标准的设计者**。

### 跨越自制阈值

随着测试、评审、反馈处理等整个开发循环都被明确编码到系统架构中后，智能体跨越了一个决定性的自制阈值：只需给定一个提示词，智能体就能完全脱离人工干预，端到端的实现。

来看看原文展示的这条全自动闭环流水线：
1. 它会先自动查验代码库现状
2. 遇到bug时自主运行应用去复现，甚至会自己录制展示错误的视频留作证据
3. 随后完成代码修复
4. 通过再次驱动应用来进行自我验证，并录制第二段成功运行的视频作为比对
5. 接着它会自己开启PR
6. 如果有人类或其他系统智能体提出审查意见，它能自行响应
7. 遇到构建报错，它会自我排查并修复
8. 除非遇到了真正涉及方向性抉择的情况，它才会按铃叫人类来判断
9. 否则他会一路走到底，最终自动合并代码

这是一个在极其严苛的结构化环境中真实跑通的软件制造闭环——从输入一段需求到最终代码合并，中间长达数小时的执行过程全由机器接管。

这种真正跑通端到端的全自动开发，已经成为全行业最顶尖的工程团队共同确立的演进目标。不仅是OpenAI，Anthropic在不久前展示的Agent Teams原型实验，也展示了相同的趋势。

智能体已经进化出自主的端到端开发能力。未来的工程师不再是帮补全某个局部函数，而是把它当做真正的外包承包方——你只需要甩出完整的需求，不再管怎么写，只定义什么算完成，然后等待它独立交付。

### 冷峻的提醒

虽然这种端到端全自动运行的前景令人兴奋，但是OpenAI紧接着给出了一个非常冷峻的提醒：

原文写道："此行为高度依赖于该代码库的具体结构和工具链，也不应假设其能普遍推广。至少目前如此，除非投入了类似的开发资源。"

这意味着这种高速运转的自制绝不能被简单地复制粘贴。它是用极高的基础设施门槛换来的。如果没有这套重金打造的环境与护栏，单纯追求让大模型自动写代码，只会制造出加速系统崩溃的数字垃圾。

这也揭示了自制的尽头绝对不是对AI撒手不管。相反，**越是高度的自制，越离不开极度严密的环境设计与工程约束**。
